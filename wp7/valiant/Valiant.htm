Valiant-tool

Introduction

 

Valiant is a transformation tool. It will transform XML files to RDF files. All XML files will be put in their own graph. A zipfile of the tool can be found on http://code.google.com/p/lod2-stack/source/browse/#svn%2Ftrunk%2Fwp7%2Fvaliant-tool .  To start the valiant tool, first unzip the valiant.zip file. In this zip file there are 2 files named valiant.properties and startup.sh. The properties file need to be modified, depending on local settings. This will be explained more in detail in the next part. When the properties file is modified, the tool can be started with startup.sh. On an unix system this can be done by using next command in the terminal:

./startup.sh

Configuring Input & Output

 

This is the main task before using the tool.

There are 2 different input configurations for the tool: The XML files can be local XML-file(s) on the disk, or remote XML-file(s) coming from a webdav. This configuration is modified in the “Inputfile” field in the properties file. Leaving this field empty will select the webdav as input configuration. When you select a path, than there will be, depending on the selected path, one or more local files transformed. Selecting the path of a single xml file will only transform this xml file. Directing the path to a directory, all xml files in this directory will be transformed.

The output configuration has 5 different possibilities. It’s possible to store the results in files on the disk, upload them to Virtuoso or upload to a webdav. The last 2 possibilities’ are to load the resulting RDF to Virtuoso and Webdav at the same time. Or write to files and upload to Virtuoso at the same time. The writing configuration (also the writing configuration mixed with Virtuoso) will write 2 different types of files on the disk. An RDF file with the transformed content. And a .graph file which contains the graphname where the file will be loaded to when uploading to Virtuoso.

The input & output configurations, and the different folders needs to be specified correctly in the valiant.properties file. Otherwise the tool won’t run. An example of how the properties file could look like:

# webdav connection settings

host = http://wp7.lod2.eu:8890

user = dav

password = wp7admin

webdavUrl = /DAV/home/wkd/rdf_sink/

 

 

 

 

# virtuoso connection settings

virtuoso.host = localhost

virtuoso.port = 1111

virtuoso.username = dba

virtuoso.password = dba

# set to true|false to immediately load the generated rdf in virtuoso

virtuoso.load = true

 

# baseURI

baseURI = http://schema.wolterskluwer.de/resource/

 

# filename regex

regex = [a-z][^\\.]*\\.xml

 

# max files to load or -1 for all files (eg. max = -1)

max = -1

 

# xsl path to wkd.xsl (eg. /my/path/to/wkd.xsl

xslPath = /home/jand/lod2-stack/wp7/xslt/wkd.xsl

 

# catalog path to catalog.xml (eg. /my/path/to/catalog.xml

catalogUrl = /home/jand/lod2-stack/wp7/catalog/catalog.xml

 

# rdf output folder

rdfFolder = /home/valiant/rdf/

 

# webdav output folder

webdavOutputFolder = /DAV/home/wkd/rdf_xml/

 

# inputfile folder (if null, transformation will go to the webdav)

inputfile =

 

# transformation mode (f = write to file, v = upload to Virtuoso, w = upload to selected webdav, fv = write to file and upload to Virtuoso, wv = upload to webdav and to virtuoso)

transformMode = f

 

# implementation mode (choose "xalan" or "saxon")

implementationMode = saxon

 

First two fields are for the connections to the webdav and Virtuoso. In the baseURI field, the wanted base URI in the graph name is specified. Filename regex can be modified to search for a certain file in the webdav (this can’t be used for searching local files). To search for only one file, “regex = berlinertab.xml” can be used for example. Next field defines the maximum amount of files you want to search in the webdav. Again this is only applied to webdav files, there is no limit for the amount of local files as input. In the next two fields, the absolute path to the xslt and catalog file will be specified. Relative paths are also allowed. The given RDF folder will contain all files when selecting a writing output configuration. Make sure that this directory exists before you run the tool. The directory might be empty at start. But if the directory doesn’t exist at startup, the tool won’t work properly. Next is the output folder for the webdav. In the inputfile field the input configuration is defined, as described above. The output configuration is defined in the transformation mode field. All 5 possibilities are described in the comment line. Last is what API is used for the transformation of XML to RDF.

Parallellism

 

Running more than 1 instance of ./startup.sh at the same time is possible in Valiant. To reach this parallelism, a blanco .RDF-file with the same name as the input file is made for every input file, right after reading this input file, independent of the output configuration. But depending on this configuration, the RDF-file will stay empty or will contain the content of the transformation.

Before transforming will take place, Valiant first checks if there is a matching .RDF-file, if that’s not the case, it will start the transformation. In this way, it’s almost impossible to have 2 processes transforming the same file. Like this, parallelism is reached.

Running 2 (or more) instances can be done with next command in the terminal:

                ./startup.sh & ./startup.sh

For more instances it’s possible to add as many & ./startup.sh as wanted. Watch out that more parallelism will only give a performance gain if the hardware specifications are good enough.

SQL Script

 

Using SQL Script is another implementation to upload to virtuoso. To use this, the created .graph files described above are necessary. A disadvantage of this approach is therefor that it’s required to use the write to files configuration first. For every XML input file, 2 other files are created this way.

Before running the script, load.sql needs to be modified. load.sql should also be in the zip file. When opening load.sql, the content should be something like this:

ld_dir('/home/wp7/rdf', '%.rdf', NULL);

Only thing that needs to be modified is the '/home/wp7/rdf' –part. This should be redirected to the chosen RDF folder you specified in the properties file. Also the virtuoso.ini file needs to be modified before this SQL Script can work. This file can be found in the Virtouso directory. In this file, add the absolute path of the RDF folder to the “DirsAllowed” parameter. Now it’s possible to run the script with next command.

isql-vt 1111 <USERNAME> <PASSWORD> load.sql

Isql-vt is part of the Virtuoso distribution on Ubuntu. So no further installation is required.

Test 1

First test was transforming 1000 files from the webdav into all 5 different output modes with no parallelism or SQL Script. This leaded to the following results:

Write to file

55,083s

Upload to webdav

34 min 24,677s

Upload to Virtuoso (Jena)

51 min 36,278s

Write to file + Upload in Virtuoso

49 min 18,336s

Upload to Virtuoso and webdav

88 min 12,160s

 

Conclusion:

Uploading to Virtuoso with the Jena API is the slowest configuration in the tool. Writing to files on the disk is very fast and uploading to a webdav is slow.

Test 2

To improve the performance we implemented a parallelism possibility as described above in the introduction. In the next test, 2 processes run at the same time and gave next results:

Write to file

33,772s

Upload to webdav

17 min 13,882s

Upload to Virtuoso (Jena)

32 min 34,712s

Write to file + Upload in Virtuoso

32 min 52,303s

Upload to Virtuoso and webdav

45 min 6,953s

 

Conclusion:

This is as expected almost twice as fast as in test 1. Running more processes at the same time can lead to even more performance gain. (Need to be verified in the future.) But the amount of parallel processes is dependent on the hardware specifications.

Test 3

Next we implemented another implementation to upload to Virtuoso. And see if this increases the performance of the tool. We wrote an SQL Script. To use this, first all files need to be written as files on the disk. This test gave next results:

Write to file (2 processes parallel)

SQL Script

40,582s

0,252s

Conclusion:

Jena takes approximate 50 minutes for uploading to Virtuoso. With SQL Script it can be done in less than a minute. Disadvantage of this approach is that it’s required to write extra files on the disk first. Although these disadvantages are ignorable compared with the performance gain.

 

 

Appendix

 

The used API’s and frameworks in this application are:

Spring Framework. (http://www.springsource.org/) 

Saxon API is used for the XML<->RDF transformation. (http://saxon.sourceforge.net/)

Xalan API is another implementation for this transformation. (http://xml.apache.org/xalan-j/)

Jena API is used for uploading RDF-files into Virtuoso. (http://jena.sourceforge.net/)

Sardine API is used for writing/reading the webdav. (http://code.google.com/p/sardine/)
